============================================================
SECTION 1 - EXECUTIVE SUMMARY
============================================================

1. Intended relationship
Verified from code and migrations: the intended Coach <-> Client relationship is anchored by a row in public.subscriptions. The subscription links one member/client to one coach and usually one coach package. Operational coaching features should unlock only when the subscription is active. After payment is confirmed, a coach_member_threads row should exist for the subscription, and coach_messages should hold the shared relationship conversation.

2. Current implementation
Verified from migration:
- Base roles are member, coach, seller only. Admin is intentionally separate through app_admins.
- subscriptions exists from the initial schema and is extended by later migrations.
- coach_packages, coach_availability_slots, coach_reviews, coach_member_threads, coach_messages, weekly_checkins, progress_photos, coach_client_records, coach_client_notes, program templates, habits, resources, bookings, Paymob orders, Paymob transactions, payout accounts, payout batches/items, admin audit events, and admin RPCs exist in repo migrations.
- Paymob test-mode backend flow exists in repo: create-coach-paymob-checkout Edge Function creates a subscription and coach_payment_orders row, creates a Paymob intention, and returns hosted checkout details. paymob-transaction-callback verifies HMAC and calls process_coach_paymob_callback. Successful verified callbacks activate the subscription, create/reuse the thread, and create/reuse a coach payout item.
- Legacy/manual receipt flow still exists and is feature-flagged in Flutter.

3. What appears complete
Verified from code/migration:
- Relationship anchor: subscriptions.
- Active-only messaging through send_coaching_message.
- Active-only member check-ins through submit_weekly_checkin.
- Coach check-in feedback creates a coach_messages row with message_type = checkin_feedback.
- Idempotent thread creation via ensure_coach_member_thread.
- Paymob test-mode tables, webhook processing, payout item creation, and admin settlement RPCs are present in repo.
- Flutter has member marketplace, coach details, package checkout, My Coaching, messages, check-ins, privacy settings, coach workspace, coach billing, resources, programs, onboarding flows, calendar, and admin dashboard screens/routes.

4. What appears incomplete
Verified from code search:
- No dedicated member-facing resources screen was found, even though resource assignment backend and coach UI exist.
- No dedicated member-facing booking/session screen was found, even though booking backend allows member reads and self-booking hooks.
- Habit assignment backend exists and coach can assign habits, but a member habit tracking UI for those coach-assigned habits was not found.
- Progress photo storage bucket was not found; progress_photos table stores storage_path, but no dedicated progress photo bucket/policies were found in migrations.
- Some admin/Paymob functionality is implemented in repo but not live-verified.
- Deno Edge Function tests are present for Paymob callback but Deno was not available in prior local verification, so Edge test execution is not confirmed.

5. What appears risky
Verified from migration:
- Initial subscriptions RLS allows member or coach updates on subscriptions. Later critical transitions use RPCs, but direct update policies may still be too broad unless later migrations tighten them elsewhere. This is high-risk because subscription status should not be directly mutable by normal clients.
- Legacy coach approval RPC verify_coach_payment still activates subscriptions for manual flow. Flutter hides it for Paymob rows, but backend still allows coach activation for manual receipts.
- apply_coach_onboarding_flow calls ensure_coach_member_thread and therefore fails if used on a non-active subscription. Its own pipeline update fallback mentions pending_payment, but thread creation requires active.
- assign_program_template_to_client and assign_client_habits only verify coach owns the subscription, not that subscription.status = active. This allows assignment to pending/paused/cancelled relationships unless product allows that.
- coach_bookings can be created for active or paused subscriptions; product decision needed.

6. What is likely to crash at runtime
Likely/verified risks:
- If migrations 34/35 are not deployed, Flutter admin dashboard and Paymob payment status providers will call missing RPCs/tables.
- If PAYMOB_* Edge env vars are missing, create-coach-paymob-checkout and callback processing will fail.
- If coach-resources or coach-payment-receipts buckets are missing live, upload flows fail.
- If progress photo upload is added without storage policy/bucket parity, photo upload will fail.
- If an active subscription lacks a thread, member message route may have no thread; check-in submission can recover by calling ensure_coach_member_thread.

7. What should be fixed first
Priority:
1. Tighten subscriptions direct update RLS and force state changes through RPCs/Edge Functions.
2. Verify/deploy Paymob migrations and Edge Functions, then run an end-to-end test-mode payment.
3. Ensure active subscription -> thread creation is recoverable and monitored.
4. Decide and enforce pending/paused/cancelled behavior for programs, habits, resources, bookings, messages, and check-ins.
5. Add missing member-facing resources, sessions, and habit tracking views or hide coach assignment surfaces until client consumption exists.

8. Production readiness
Not production-ready yet. The repo has a strong backend design for the core relationship and Paymob test-mode settlement, but live deployment is not verified, some RLS remains broad, legacy/manual payment activation still exists, and several coach tools are backend/coach-side only without complete member-facing consumption.

Payment gateway/Paymob
Verified from Edge Functions:
- Receives Create Intention API calls from create-coach-paymob-checkout using server-side PAYMOB_SECRET_KEY_TEST.

- Sends callback/webhook to paymob-transaction-callback.
- Must not be trusted unless HMAC is verified with PAYMOB_HMAC_SECRET_TEST.
- Does not pay coach directly in MVP. Money goes to GymUnity merchant account.

System/backend jobs/functions

SECTION 3 - FULL RELATIONSHIP LIFECYCLE
============================================================

1. Coach signs up
- User-visible action: register/select coach role.
- Screen/route: role selection/register/onboarding screens.
- Frontend: Auth and User repositories.
- Tables: users, profiles(role_id=coach), coach_profiles later.
- Resulting state: profile exists; onboarding_completed false until profile saved.
- Missing/risk: admin is not selectable, which is correct.

2. Coach completes onboarding
- Screen/route: /coach-onboarding, /coach-profile.
- Button/control: save profile.
- Repository: CoachRepositoryImpl.upsertCoachProfile.
- Tables: coach_profiles upsert, profiles.onboarding_completed = true.
- Resulting state: public coach profile can be listed.
- Possible errors: missing columns handled by schema-compatible write fallback.

3. Coach creates/edits profile
- Screen/route: /coach-profile.
- Backend: direct coach_profiles upsert, profiles update.
- RLS: coach_profiles_manage_own.
- Data visibility: coach_profiles are broadly readable; profiles are own-only, so public profile RPC provides name fallback.

4. Coach creates package/offer
- Screen/route: /packages, /add-package.
- Button/control: create/edit offer, save offer, publish/draft/archive controls.
- Repository: saveCoachPackage, deleteCoachPackage.
- Tables: coach_packages.
- States: draft, published, archived through visibility_status; is_active mirrors published in UI.
- Possible errors: package linked to subscriptions is archived instead of deleted.

5. Coach publishes package/offer
- Screen: CoachPackageEditorScreen or CoachPackagesScreen.
- Backend: coach_packages.visibility_status='published', is_active=true.
- Client discoverability: list_coach_public_packages filters active/published.

6. Client discovers coach
- Screen/route: /coaches.
- Repository: CoachRepositoryImpl.listCoaches.
- RPC: list_coach_directory_v2 with fallback to list_coach_directory.
- Data: public coach profile, starting package price, filters by specialty/city/language/gender/budget.
- Empty/error states exist in screen.

7. Client views coach details
- Screen/route: /coach-details.
- Repository: getCoachDetails.
- RPCs/tables: get_coach_public_profile, listCoachPackages/list_coach_public_packages, availability, reviews.
- Button: view packages/start coaching routes to /subscription-packages.

8. Client views packages
- Screen/route: /subscription-packages.
- Data: coach packages, package preview, intake/payment controls.
- Button: Paymob enabled -> "Start secure payment" / "Pay with Paymob"; legacy -> "Start paid checkout" / "Submit request".
- Config: ENABLE_COACH_PAYMOB_PAYMENTS controls Paymob path; ENABLE_COACH_MANUAL_PAYMENT_PROOFS controls legacy proof UI.

9. Client starts checkout
- Paymob path:
  - Frontend method: CoachPaymentRepositoryImpl.createPaymobCheckout.
  - Edge Function: create-coach-paymob-checkout.
  - Tables: subscriptions insert with status=checkout_pending, checkout_status=checkout_pending, payment_gateway=paymob; coach_payment_orders insert status=created then pending.
  - Paymob API: /v1/intention/.
  - Result: returns payment_order_id, subscription_id, paymob_client_secret, paymob_public_key, checkout_url, amount, currency, mode=test.
- Legacy path:
  - Repository: CoachRepositoryImpl.requestSubscription.
  - RPC: create_coach_checkout.
  - Tables: subscriptions, coach_checkout_requests.

10. Client pays
- Paymob path: Flutter opens hosted checkout_url using url_launcher/external link service. Flutter does not activate subscription.
- Legacy path: client submits manual reference/receipt in My Coaching.

11. Payment succeeds
- Paymob source of truth:
  - Function: paymob-transaction-callback.
  - HMAC: verifyPaymobTransactionHmac.
  - RPC: process_coach_paymob_callback.
  - Validation: mode=test, amount, currency, order lookup, subscription existence, duplicate transaction id.
  - State: coach_payment_orders.status=paid, subscriptions.status=active, checkout_status=paid.
  - Side effects: ensure_coach_member_thread, create_pending_coach_payout_for_payment, notifications to coach/member/admin.
- Legacy source:
  - RPC verify_coach_payment by coach activates subscription and creates thread.
  - This conflicts with Paymob business model if exposed for Paymob rows.

12. Payment fails
- Paymob:
  - order status failed/refunded, subscription checkout_status failed/refunded, status remains checkout_pending unless already paid.
  - member/admin notification created.
- Legacy:
  - coach failPayment updates receipt failed and subscription checkout_status failed.

13. Subscription becomes active
- Backend: Paymob verified callback or legacy verify_coach_payment.
- Result: status=active, checkout_status=paid, starts_at/ends_at/current_period values set, thread created.
- Operational unlock: messages/check-ins active-only enforced in RPCs.

14. Thread is created
- RPC: ensure_coach_member_thread(subscription_id).
- Idempotency: unique subscription_id, conflict-safe insert, prevents duplicate system message.
- Creates coach_member_threads and one system coach_messages row.
- Missing/risk: no background job found; relies on payment callback/check-in/admin repair.

15. Client sends message
- Screen/route: /member-messages -> /member-thread.
- Repository: MemberRepositoryImpl.sendCoachingMessage.
- RPC: send_coaching_message.
- Enforcement: member/coach role only, participant thread, active subscription, non-empty content.
- Tables: coach_messages insert, coach_member_threads last preview update.

16. Coach sends message
- Screen: coach client workspace message area.
- Repository: CoachRepositoryImpl.sendCoachMessage.
- RPC: send_coaching_message.
- Same enforcement as member.

17. Client submits check-in
- Screen/route: /member-checkins.
- Repository: MemberRepositoryImpl.submitWeeklyCheckin.
- RPC: submit_weekly_checkin.
- Tables: weekly_checkins upsert, progress_photos replace for checkin, member_weight_entries insert if weight, member_body_measurements insert if waist.
- Enforcement: member role, owns subscription, status active, one row per subscription/week.

18. Coach reviews check-in
- Screens: /coach-checkins or /coach-client-workspace.
- Repository: CoachRepositoryImpl.submitCheckinFeedback.
- RPC: submit_coach_checkin_feedback.
- Tables: weekly_checkins.coach_reply, coach_messages checkin_feedback.
- Enforcement: coach role, active subscription, non-empty feedback.

19. Coach sends feedback
- Same as step 18.
- Notification: member receives "Coach feedback received".

20. Coach assigns program
- Screens: /coach-program-library, /coach-client-workspace.
- Repository: assignProgramTemplate.
- RPC: assign_program_template_to_client.
- Tables: workout_plans, workout_plan_days, workout_plan_tasks.
- Side effect: archives existing active coach-authored plan for that member/coach.
- Risk: active subscription is not enforced by the RPC; it only verifies coach owns the subscription.

21. Coach assigns resource
- Screens: /coach-resources, /coach-client-workspace, onboarding flow.
- Repository: uploadCoachResource/saveCoachResource/assignResourceToClient.
- RPCs/tables/storage: coach-resources bucket, coach_resources, coach_resource_assignments.
- Notification: member receives "New coaching resource".
- Missing: member-facing assigned resources screen not found.

22. Coach schedules session/booking
- Screen: /coach-calendar and coach client workspace.
- Repository: createBooking/updateBookingStatus.
- RPCs: create_coach_booking, update_coach_booking_status.
- Tables: coach_session_types, coach_bookings.
- Eligibility: active or paused subscriptions.
- Missing: dedicated member booking screen not found.

23. Client manages privacy visibility
- Screen/route: /member-coach-visibility from My Coaching.
- Repository: CoachMemberInsightsRepositoryImpl.upsertVisibilitySettings.
- RPC: upsert_coach_member_visibility.
- Tables: coach_member_visibility_settings, coach_member_visibility_audit.
- Result: toggles control coach access to AI plan/adherence/progress/nutrition/product/purchase insights.

24. Coach views allowed insights
- Screen/route: /coach-member-insights.
- Repository: getMemberInsight/listClientInsightSummaries.
- RPCs: get_coach_member_insight, list_coach_member_insight_summaries.
- Enforcement: active subscription and coach is subscriber coach. No settings row returns header only.

25. Subscription is paused
- Frontend: MemberRepositoryImpl.pauseSubscription calls pause_coach_subscription.
- Migration defining pause_coach_subscription was not inspected in the primary relationship migrations; repo search shows Flutter expects it.
- Behavior found elsewhere: create_coach_booking allows active or paused; messages/check-ins require active only.
- Current uncertainty: pause/resume full backend state machine not fully verified.

26. Subscription is resumed
- Not clearly found in inspected code. Unclear/missing.

27. Subscription is completed/cancelled
- Status values exist. Admin cancel unpaid checkout exists for unpaid Paymob orders. Full member/coach cancellation UX not clearly found.

28. Admin pays coach payout
- Screen/route: /admin-dashboard, Payouts tab.
- Repository: AdminRepositoryImpl.markPayoutPaid.
- RPC: admin_mark_payout_paid.
- Requirements: permission payouts.mark_paid, external_reference required, status ready/processing.
- Tables: coach_payouts updated, admin_audit_events inserted, notifications to coach.
- No money transfer triggered.

29. Refund/dispute/manual correction
- Paymob transaction status can set refunded/voided and order refunded/failed if not already paid.
- Full refund/dispute automation not implemented.
- Admin reconcile currently marks needs_review, not paid.

SECTION 6 - STATE MACHINES
============================================================

Package visibility/status
- Flutter states: draft, published, archived.
- Database: coach_packages.visibility_status in later migrations, is_active boolean.
- Valid transitions: draft -> published, published -> archived/draft, archived -> draft/published if coach restores.
- UI labels: package.status chips.
- Enforcement: public package RPC filters published/is_active.
- Risk: older code may use is_active only.

Subscription status
- Found database statuses: checkout_pending, pending_payment, pending_activation, active, paused, completed, cancelled. Initial schema default was active.
- Paymob create checkout: checkout_pending.
- Paymob success: active.
- Paymob failure: status remains checkout_pending unless manually cancelled.
- Manual legacy: checkout_pending -> active via coach verify; failed changes checkout_status only.
- Invalid/risky combinations:
  - active but payment not paid (possible through legacy/manual or direct broad RLS).
  - paid Paymob order but subscription not active (admin dashboard alert).
  - active but no thread (admin repair exists).

Checkout/payment status
- Found checkout_status values: not_started, checkout_pending, payment_pending, submitted, receipt_uploaded, under_verification, paid, failed, refunded.
- Legacy receipt statuses: awaiting_payment, submitted, receipt_uploaded, under_verification, activated, failed.
- Paymob order statuses: created, pending, paid, failed, cancelled, refunded, partially_refunded.
- Mismatch risk: old migration 33 did not include payment_pending; migration 34 adds it.

Paymob payment order status
- created: order inserted before Paymob intention response.
- pending: Paymob intention created, checkout open.
- paid: verified successful callback.
- failed: callback failure, validation mismatch, intention creation failure.
- cancelled/refunded/partially_refunded: scaffolded.
- Backend enforcement: process_coach_paymob_callback validates mode, amount, currency, order, subscription.

Payout status
- pending, on_hold, ready, processing, paid, failed, cancelled.
- create_pending_coach_payout_for_payment sets ready if hold_days=0, on_hold if >0.
- admin_mark_payout_paid requires ready or processing in migration 35.
- Legacy payout_status values also exist on adapted coach_payouts: eligible, held, paid etc.
- Risk: two status systems exist (status and payout_status) for backward compatibility.

Thread/message state
- Thread missing: normal before active/payment.
- Thread created: one per subscription via unique(subscription_id).
- Message types: text, checkin_prompt, checkin_feedback, system.
- send_coaching_message only allows text and active subscription.
- check-in feedback inserts message_type=checkin_feedback.
- Risk: direct insert policy for coach_messages was dropped in migration 33; RPC is intended path.

Check-in state
- Not submitted: no weekly_checkins row.
- Submitted: row exists with coach_reply null.
- Reviewed/feedback sent: coach_reply not null and feedback message inserted.
- Overdue: inferred in action item queries by no recent checkin; no explicit status column.
- Valid transitions: none -> submitted, submitted -> same-week update, submitted -> feedback.
- Backend: active-only for submit/feedback.

Booking state
- Found: scheduled, confirmed, completed, cancelled, rescheduled, no_show.
- create_coach_booking starts scheduled.
- update_coach_booking_status can set any allowed status by participant.
- Risk: state transitions are not strict; any participant can update to completed/no_show/cancelled.

Privacy visibility state
- No settings row: coach receives header only.
- All toggles false: settings/audit exist, no insight details.
- Partial sharing: only selected categories returned.
- All sharing: all categories returned.
- Revoked all: audit change_type revoked_all.
- Backend enforcement: coach insight RPC requires active subscription and checks toggles.

Dangerous combinations found/guarded
- subscription active but payment not paid: possible via legacy/manual direct flows; Paymob model avoids if used exclusively.
- payment paid but subscription not active: admin alert/RPC detects.
- subscription active but no thread: admin repair + ensure in check-in.
- thread exists for cancelled subscription: possible historically; messaging/check-ins still blocked by active check.
- payout paid for unpaid order: unique payout item ties paid order creation; admin pay acts on payout, but direct data integrity should be audited.
- coach approves Paymob payment: UI hides actions for Paymob, but verify_coach_payment legacy remains.
- client sees messages before activation: send RPC blocks; direct read of preexisting thread possible if thread exists, but thread creation blocks non-active.
- check-in before activation: submit_weekly_checkin blocks.
- privacy insights without consent: get_coach_member_insight returns null sections unless toggles set.

SECTION 7 - PAYMENT AND PAYOUT FLOW
============================================================

A. Old/manual payment proof model
- Screens: SubscriptionPackagesScreen legacy path, MySubscriptionsScreen proof upload, CoachBillingScreen review queue.
- Buttons: Start paid checkout, Submit request, Submit payment proof, Approve payment, Mark failed/Needs follow-up.
- RPCs/functions: create_coach_checkout, submit_coach_payment_receipt, verify_coach_payment, fail_coach_payment, list_coach_payment_queue.
- Tables: subscriptions, coach_checkout_requests, coach_payment_receipts, coach_payment_audit_events, notifications, coach_member_threads after approval.
- Status transitions:
  - checkout_pending/checkout_pending -> submitted/receipt_uploaded -> active/paid on coach verify.
  - checkout_pending -> failed checkout_status on coach failure.
- Who approves payment: coach.
- Who receives money: unclear/manual outside app.
- Security risk: conflicts with Paymob platform-guarantor model.
- Current status: still present and feature-flagged; should remain legacy only.

B. Paymob/admin settlement model
- Screens: SubscriptionPackagesScreen Paymob path, MySubscriptionsScreen Paymob status, CoachBillingScreen status only, AdminDashboard settlement.
- Buttons: Start secure payment, Pay with Paymob, refresh/retry, admin mark payout paid.
- Edge Functions: create-coach-paymob-checkout, paymob-transaction-callback, paymob-payment-response, sync-paymob-payment-status, admin-payment-settings.
- RPCs: process_coach_paymob_callback, create_pending_coach_payout_for_payment, get_coach_payment_order_status, admin_* payment/payout RPCs.
- Tables: subscriptions, coach_payment_orders, coach_payment_transactions, coach_payouts, coach_payout_items, admin_audit_events, notifications.
- Status transitions:
  - subscription checkout_pending + order created/pending.
  - verified callback success -> order paid, subscription active/paid, thread created, payout item created.
  - callback failure/mismatch -> order failed, checkout_status failed, no activation.
- Who approves payment: Paymob verified HMAC callback processed by GymUnity backend.
- Who receives money: GymUnity Paymob merchant account.
- Who pays coach: admin manually outside app; app records payout only.
- Webhook delayed: Flutter shows pending/confirming and refreshes; no activation until callback.
- Duplicate webhook: unique paymob_transaction_id causes duplicate=true and no duplicate payout item/notifications after first insert.
- Amount/currency mismatch: order/subscription set failed if not already paid; processing_result records mismatch.
- User closes checkout: order remains pending; subscription checkout_pending.
- Existing subscription: create function rejects duplicate open relationship by package or coach, except failed Paymob retry can reuse failed row.

Answers
1. Which model is currently active?
Both are implemented. Flutter uses Paymob only when ENABLE_COACH_PAYMOB_PAYMENTS=true; manual proof remains available when ENABLE_COACH_MANUAL_PAYMENT_PROOFS=true and for non-Paymob rows.
2. Which should remain?
Paymob/admin settlement should be primary. Manual should remain hidden behind legacy flag only.
3. Old screens/buttons to hide/deprecate:
Submit payment proof, approve payment, fail payment, manual payment rail selector for Paymob-enabled packages/orders.
4. Required changes:
Deploy/verify Paymob migrations/functions, tighten subscription RLS, disable manual proofs by default for new Paymob rollout, ensure admin settlement works live.

SECTION 20 - FINAL SYSTEM DIAGRAMS
============================================================

1. Happy path

Coach creates package
  -> package published
  -> client views coach/package
  -> client starts Paymob checkout
  -> create-coach-paymob-checkout creates subscription + payment order
  -> Paymob hosted checkout
  -> verified callback
  -> process_coach_paymob_callback
  -> coach_payment_orders.status = paid
  -> subscriptions.status = active / checkout_status = paid
  -> ensure_coach_member_thread
  -> client and coach messages/check-ins unlock
  -> create_pending_coach_payout_for_payment
  -> admin later marks payout paid

2. Payment failure

Client checkout
  -> Paymob failed/rejected callback
  -> HMAC verified
  -> order.status = failed
  -> subscription.checkout_status = failed
  -> no active subscription
  -> no thread unlock
  -> client can retry or admin can cancel unpaid checkout

3. Webhook delay

Flutter checkout success screen
  -> app says "confirming"
  -> subscription still checkout_pending
  -> messages/check-ins locked
  -> Paymob callback arrives later
  -> backend verifies HMAC
  -> subscription active
  -> thread created
  -> app refresh shows active

4. Check-in loop

Active subscription
  -> client submits weekly check-in
  -> weekly_checkins upsert
  -> coach notification
  -> coach reviews check-in
  -> submit_coach_checkin_feedback
  -> weekly_checkins.coach_reply updated
  -> coach_messages checkin_feedback inserted
  -> member notification

5. Privacy loop

Active subscription
  -> member opens privacy settings
  -> toggles sharing categories
  -> upsert_coach_member_visibility
  -> visibility audit row
  -> coach insight RPC checks toggles
  -> sections unlock/lock
  -> revoke all returns header only

6. Admin payout loop

Paid Paymob order
  -> payout item created once
  -> payout batch pending/ready/on_hold
  -> admin transfers money outside GymUnity
  -> admin opens /admin-dashboard
  -> mark payout paid with external reference
  -> coach_payouts.status = paid
  -> admin_audit_events row
  -> coach notification
